## Решение: `(model, is_list)` dispatch pattern

**Контекст:** Большинство endpoint-ов Journal возвращают списки объектов, но ранняя версия валидатора исходила из того, что ответ почти всегда словарь под одну Pydantic-модель. Из-за этого реальные list-endpoint-ы валидировались некорректно и давали ложные ошибки.

**Варианты которые рассматривались:**

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Оставить одну модель на endpoint без информации о типе контейнера | Самая простая карта соответствий | Не совпадает с реальной формой ответа API |
| Хранить отдельные list-wrapper модели для каждого endpoint | Формально единообразно для Pydantic | Много лишних обёрток, хуже читаемость |
| Хранить `(ModelClass, is_list)` в `MODELS` и валидировать каждый элемент списка отдельно | Совпадает с реальным API, простая диспетчеризация | Карта `MODELS` становится более многословной |

**Выбрали:** Хранить tuple `(ModelClass, is_list)` в `MODELS` и валидировать list endpoint-ы поэлементно.

**Почему:** Такой формат явно фиксирует форму ответа каждого endpoint-а и убирает ложное предположение, что все ответы словари. Для list endpoint-ов используются item-модели вроде `AverageProgressItem`, `AttendanceItem`, `LeaderboardItem`, а не вымышленные dict-wrapper схемы.

**Что теряем:** Карта `MODELS` стала более подробной, и для каждого endpoint-а теперь нужно поддерживать не только модель, но и признак контейнера.

**Когда пересмотреть:** Если появится единый слой декларации схем endpoint-ов или генерация моделей напрямую из реального OpenAPI/JSON Schema.

## Решение: `hashFiles()` → step output для issue gating

**Контекст:** В workflow GitHub Actions логика создания Issue изначально опиралась на `hashFiles()`, но эта функция смотрит на состояние файлов в checkout на момент запуска job, а не на runtime-артефакты, созданные внутри шага. В результате Issue создавался почти на каждом прогоне.

**Варианты которые рассматривались:**

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Оставить `hashFiles()` | Короткое условие в YAML | Проверяет не тот источник истины |
| Проверять наличие файла в отдельном shell step | Работает в runtime | Нужно вручную пробрасывать состояние между шагами |
| Выставлять `validation_failed=true` через `GITHUB_OUTPUT` внутри шага pipeline | Условие опирается на фактический результат шага | В workflow появляется немного shell-логики |

**Выбрали:** Внутри шага писать `validation_failed=true` в `GITHUB_OUTPUT` и использовать это значение в `if:` следующего шага.

**Почему:** Это связывает условие создания Issue с реальным результатом текущего запуска pipeline, а не с файлами, лежащими в репозитории на момент checkout.

**Что теряем:** YAML стал чуть менее декларативным, потому что часть состояния теперь формируется shell-командой внутри шага.

**Когда пересмотреть:** Если workflow будет вынесен в composite action или появится более явный статус шага без ручной записи в output.

## Решение: `data/validation_issue.md` исключён из репозитория

**Контекст:** Файл `data/validation_issue.md` был закоммичен в репозиторий, поэтому любые проверки, завязанные на его наличие, видели его всегда. Это усиливало ложные срабатывания и ломало ожидаемую семантику "файл есть только если текущая валидация упала".

**Варианты которые рассматривались:**

| Вариант | Плюсы | Минусы |
|---------|-------|--------|
| Оставить файл в репозитории | Удобно видеть пример формата | Ломает runtime-проверки и засоряет repo state |
| Переименовать файл и продолжить коммитить его | Сохраняет артефакт в истории | Не решает корневую проблему |
| Убрать файл из индекса и добавить в `.gitignore` | Файл существует только при реальном сбое текущего запуска | В репозитории больше нет постоянного примера этого файла |

**Выбрали:** Сделать `git rm --cached` и добавить `data/validation_issue.md` в `.gitignore`.

**Почему:** Файл ошибки должен быть временным runtime-артефактом CI runner, а не постоянной частью репозитория.

**Что теряем:** Нельзя просто открыть репозиторий и увидеть актуальный `validation_issue.md`, потому что он больше не хранится в git.

**Когда пересмотреть:** Если понадобится хранить шаблон Issue в репозитории отдельно от runtime-файла, например в `docs/` или `.github/`.
